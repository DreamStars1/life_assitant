# 小助手项目需求规格说明书（PRD）

> 版本：v1.0 | 日期：2026-06-19 | 状态：草案

---

## 一、文档说明

### 1.1 目的

本文档定义"小助手"项目的完整需求，包括功能性需求和非功能性需求，作为后续设计、开发、测试的基准依据。

### 1.2 范围

覆盖以下能力域：微信 Bot 对话、待办管理、日程管理、生活记录、AI 分析、伴侣协作、推送通知、PWA 管理界面。

### 1.3 术语

| 术语 | 说明 |
|------|------|
| 用户 | 系统的注册使用者（情侣中的一方） |
| 伴侣/Partner | 通过 partner_id 关联的另一位用户 |
| Bot | 微信对话机器人，通过 iLink 接入 |
| Agent | LLM 驱动的意图识别与技能路由层 |
| PWA | Progressive Web App，可安装到手机主屏幕 |

## 二、功能性需求

### FR-1 用户认证与会话

#### FR-1.1 注册与登录

| 编号 | 需求描述 |
|------|----------|
| FR-1.1.1 | 系统 SHALL 支持用户名 + 密码注册（无需邮箱验证，最简单流程） |
| FR-1.1.2 | 系统 SHALL 使用 JWT Bearer Token 进行 API 认证，Token 有效期 7 天 |
| FR-1.1.3 | 系统 SHALL 支持绑定微信 ID（iLink user_id），绑定后可通过微信对话交互 |

#### FR-1.2 伴侣关联

| 编号 | 需求描述 |
|------|----------|
| FR-1.2.1 | 用户 SHALL 能通过邀请码与伴侣建立关联：A 生成邀请码 → B 在 PWA 中输入 → partner_id 互绑 |
| FR-1.2.2 | 关联后所有待办和日程自动全量共享（建立伴侣关系即共享全部，无需逐条设置） |

---

### FR-2 微信 Bot 对话

#### FR-2.1 消息收发

| 编号 | 需求描述 |
|------|----------|
| FR-2.1.1 | 系统 SHALL 通过 iLink 长轮询（35s 超时）持续获取微信消息 |
| FR-2.1.2 | 系统 SHOULD 在收到消息后发送"正在输入"状态（首版降级，后续补上） |
| FR-2.1.3 | 系统 SHALL 支持文本消息的收发，未来可扩展图片/语音 |

#### FR-2.2 意图识别

| 编号 | 需求描述 |
|------|----------|
| FR-2.2.1 | Agent 层 SHALL 通过 LLM 解析用户自然语言，识别以下意图：创建待办、查看待办、完成待办、创建日程、查看日程、记录饮食、记录运动、记录工作、数据分析、普通聊天 |
| FR-2.2.2 | 意图识别 SHALL 输出结构化 JSON（intent + params），用于后续技能路由 |
| FR-2.2.3 | 系统 SHALL 支持自然语言日期解析（如"明天下午3点"→ 具体 datetime），时区默认 Asia/Shanghai |

#### FR-2.3 对话记忆

| 编号 | 需求描述 |
|------|----------|
| FR-2.3.1 | 系统 SHALL 将对话记录持久化到 conversations 表 |
| FR-2.3.2 | 系统 SHOULD 在对话中携带最近 N 轮上下文（通过 Redis 会话管理） |

---

### FR-3 待办管理

#### FR-3.1 基础 CRUD

| 编号 | 需求描述 |
|------|----------|
| FR-3.1.1 | 用户 SHALL 能创建待办，字段包含：标题（必填）、描述、截止时间、提醒时间、优先级、分类 |
| FR-3.1.2 | 优先级 SHALL 支持四级：low / medium / high / urgent |
| FR-3.1.3 | 状态 SHALL 支持四态流转：pending → in_progress → done；任意状态可 → cancelled |
| FR-3.1.4 | 用户 SHALL 能按状态、分类、截止时间筛选待办列表 |
| FR-3.1.5 | 用户 SHALL 能通过自然语言模糊匹配完成待办（如"把买菜的待办完成了"） |

#### FR-3.2 分配

| 编号 | 需求描述 |
|------|----------|
| FR-3.2.1 | 用户 SHALL 能将待办分配给伴侣（assigned_by 字段记录分配者） |
| FR-3.2.2 | 被分配者 SHALL 收到推送通知 |

#### FR-3.3 提醒

| 编号 | 需求描述 |
|------|----------|
| FR-3.3.1 | 系统 SHALL 每分钟扫描待到期/已过期的提醒任务，通过 Web Push 推送 |
| FR-3.3.2 | 提醒 SHALL 包含待办标题和截止时间 |

---

### FR-4 日程管理

#### FR-4.1 基础 CRUD

| 编号 | 需求描述 |
|------|----------|
| FR-4.1.1 | 用户 SHALL 能创建日程，字段包含：标题（必填）、描述、预计开始时间 planned_start_time（必填）、预计结束时间 planned_end_time、分类、提前提醒分钟数 |
| FR-4.1.2 | 用户 SHALL 能按时间范围查询日程列表（?start=&end=） |

#### FR-4.2 重复规则

| 编号 | 需求描述 |
|------|----------|
| FR-4.2.1 | 系统 SHALL 支持 JSON 格式的重复规则，包含 freq（daily/weekly/monthly/yearly）和 by_weekday（如 [1,3,5] 表示周一三五），Schema 向下兼容未来扩展 |
| FR-4.2.2 | `recurrence` 字段 SHALL 存储 JSON（如 `{"freq": "weekly", "by_weekday": [1,3,5]}`），不引入完整 RRULE 解析库 |

#### FR-4.3 共享

| 编号 | 需求描述 |
|------|----------|
| FR-4.3.1 | 用户 SHALL 能将日程标记为"共享给伴侣"（is_shared=1） |
| FR-4.3.2 | 伴侣 SHALL 能通过 /api/v1/partner/events 查看对方的共享日程 |

#### FR-4.4 预计与实际

| 编号 | 需求描述 |
|------|----------|
| FR-4.4.1 | 每条日程 SHALL 包含"预计时间"（planned_start_time / planned_end_time）和"实际时间"（actual_start_time / actual_end_time），后者可为空 |
| FR-4.4.2 | 创建时 SHALL 默认填充预计时间；实际时间在事件发生后由用户补充录入 |
| FR-4.4.3 | 用户 SHALL 能在日程详情中对比查看预计与实际时间的偏差 |
| FR-4.4.4 | AI 分析 SHALL 利用预计/实际偏差数据生成时间管理建议（如"本周3次日程超时，平均延迟25分钟"） |

#### FR-4.5 日程记录

| 编号 | 需求描述 |
|------|----------|
| FR-4.5.1 | 每条日程 SHALL 支持追加多条记录（event_records），作为执行日志或笔记 |
| FR-4.5.2 | 记录内容 SHALL 包含：文本内容、图片链接（可选）、记录时间 |
| FR-4.5.3 | 用户 SHALL 能通过微信对某项日程追加记录（如"今天的会议 [日程标题] 推迟了30分钟，讨论了Q3预算"） |
| FR-4.5.4 | 日程详情页 SHALL 以时间线形式展示所有关联记录 |

#### FR-4.6 日历视图

| 编号 | 需求描述 |
|------|----------|
| FR-4.6.1 | 前端 SHALL 提供月视图、周视图、日视图三种日历展示模式 |
| FR-4.6.2 | 周视图 SHALL 以 7 列网格展示，每列包含当天的日程块，支持横向滚动查看全天时间轴 |
| FR-4.6.3 | 日视图 SHALL 以 24 小时纵向时间轴展示当天所有日程，日程块按时间位置垂直排布 |
| FR-4.6.4 | 不同分类的日程 SHALL 以不同颜色区分（如工作=蓝、个人=绿、健康=橙） |
| FR-4.6.5 | 日程块 SHALL 点击展开详情（标题、时间、预计/实际状态、最近一条记录摘要） |
| FR-4.6.6 | 日历视图 SHALL 支持左右滑动切换日期/周/月 |

---

### FR-5 生活记录

#### FR-5.1 记录类型

| 编号 | 需求描述 |
|------|----------|
| FR-5.1.1 | 系统 SHALL 支持五种记录类型：饮食（diet）、运动（exercise）、工作（work）、心情（mood）、睡眠（sleep） |
| FR-5.1.2 | 每种类型的结构化元数据 SHALL 存储在 metadata_json 字段中（JSON 格式） |

#### FR-5.2 录入方式

| 编号 | 需求描述 |
|------|----------|
| FR-5.2.1 | 用户 SHALL 能通过微信自然语言录入（如"中午吃了米饭鸡胸肉西兰花，大概650卡"） |
| FR-5.2.2 | 用户 SHALL 能通过 PWA 表单录入 |
| FR-5.2.3 | 记录 SHALL 区分 logged_at（发生时间）和 created_at（创建时间） |

#### FR-5.3 查询与统计

| 编号 | 需求描述 |
|------|----------|
| FR-5.3.1 | 用户 SHALL 能按类型和时间范围查询记录 |
| FR-5.3.2 | 系统 SHALL 提供按周/月的统计数据接口（/api/v1/life-logs/stats） |

---

### FR-6 AI 分析

#### FR-6.1 分析类型

| 编号 | 需求描述 |
|------|----------|
| FR-6.1.1 | 系统 SHALL 支持四种分析类型：每日（daily）、每周（weekly）、每月（monthly）、按需（on_demand） |
| FR-6.1.2 | 每日分析 SHALL 在每天固定时间自动生成 |
| FR-6.1.3 | 每周分析 SHALL 在每周日 20:00 自动生成并推送 |

#### FR-6.2 分析内容

| 编号 | 需求描述 |
|------|----------|
| FR-6.2.1 | AI 分析 SHALL 基于用户的生活记录、待办完成率、日程数据生成摘要和建议 |
| FR-6.2.2 | 分析结果 SHALL 包含 summary（文本摘要）和 suggestions（结构化建议列表） |
| FR-6.2.3 | 用户 SHALL 能对建议进行"采纳"操作（accepted 字段） |

---

### FR-7 推送通知

#### FR-7.1 订阅管理

| 编号 | 需求描述 |
|------|----------|
| FR-7.1.1 | 前端 SHALL 通过 Service Worker 注册 Web Push 订阅 |
| FR-7.1.2 | 订阅信息（endpoint + p256dh + auth）SHALL 持久化到 push_subscriptions 表 |
| FR-7.1.3 | 用户 SHALL 能取消推送订阅 |
| FR-7.1.4 | 推送失败时系统 SHALL 将订阅标记为 is_active=0 |

#### FR-7.2 推送类型

| 编号 | 需求描述 |
|------|----------|
| FR-7.2.1 | 系统 SHALL 支持以下推送类型：待办提醒、日程提醒、每日摘要、周报、伴侣分配通知 |
| FR-7.2.2 | 推送失败 SHALL 通过 logger.warning 记录（不做数据库持久化，减少表维护负担） |

#### FR-7.3 推送偏好

| 编号 | 需求描述 |
|------|----------|
| FR-7.3.1 | 用户 SHALL 能设置静默时段（quiet_hours_start ~ quiet_hours_end），该时段内不推送 |
| FR-7.3.2 | 用户 SHALL 能全局开关推送（push_enabled） |

### FR-8 PWA 管理界面

#### FR-8.1 页面结构

| 编号 | 需求描述 |
|------|----------|
| FR-8.1.1 | 应用 SHALL 包含以下页面：今日首页、待办列表、日程日历、数据分析、协作共享、设置 |
| FR-8.1.2 | 今日首页 SHALL 聚合展示今日待办、今日日程、快捷录入入口 |

#### FR-8.2 PWA 特性

| FR-8.2.3 | 应用 SHALL 使用 Vant 4 移动端组件库，适配手机屏幕 |

#### FR-8.3 数据可视化

| 编号 | 需求描述 |
|------|----------|
| FR-8.3.1 | 数据分析页 SHALL 使用 ECharts 渲染图表（如周运动量趋势、饮食热量分布） |

## 三、非功能性需求

### NFR-1 可用性

| 编号 | 需求描述 | 指标 |
|------|----------|------|
| NFR-1.1 | 系统可用性 SHALL 达到 99%（非关键时段允许计划内维护） | ≥ 99% |
| NFR-1.2 | 数据库 SHALL 每日自动备份，保留最近 30 天 | 每日 |
| NFR-1.3 | 系统 SHALL 提供健康检查接口（/api/v1/health），检查 MySQL、Redis、iLink、DeepSeek 连通性 | - |

### NFR-2 安全

| 编号 | 需求描述 |
|------|----------|
| NFR-2.1 | 所有外部通信 SHALL 使用 HTTPS/TLS |
| NFR-2.2 | API 认证 SHALL 使用 JWT，敏感配置（密钥、Token）SHALL 通过环境变量注入，不得硬编码 |
| NFR-2.3 | 数据库密码 SHALL 通过 .env 文件管理，.env 不得提交到版本控制 |
| NFR-2.4 | 用户数据隔离：用户 A SHALL 不能访问用户 B 的私有数据，除非通过共享授权 |
| NFR-2.5 | SQL 注入防护：所有数据库操作 SHALL 使用参数化查询（SQLAlchemy ORM） |

### NFR-3 可维护性

| 编号 | 需求描述 |
|------|----------|
| NFR-3.1 | 后端代码 SHALL 按 DDD 限界上下文组织：identity / planning / lifelog / agent / analysis / notification（每个上下文内聚 models + service + router） |
| NFR-3.2 | 数据库变更 SHALL 通过 Alembic 迁移管理 |
| NFR-3.3 | Python 代码 SHALL 通过 Black + isort + flake8 格式化与检查 |
| NFR-3.4 | TypeScript 代码 SHALL 通过 ESLint + Prettier 格式化与检查 |
| NFR-3.5 | Git 提交 SHALL 遵循 Conventional Commits 规范（feat/fix/docs/refactor/chore） |

### NFR-4 可扩展性

| 编号 | 需求描述 |
|------|----------|
| NFR-4.1 | Agent 技能 SHALL 通过 BaseSkill 抽象类注册，新增技能无需修改核心路由逻辑 |
| NFR-4.2 | 系统 SHALL 支持通过配置切换 LLM 提供商（DeepSeek / Qwen） |
| NFR-4.3 | 前端 SHALL 预留 Capacitor 配置，支持未来打包为 APK |

### NFR-5 兼容性

| 编号 | 需求描述 |
|------|----------|
| NFR-5.1 | PWA SHALL 兼容 Chrome、Edge、Safari 主流移动浏览器 |
| NFR-5.2 | Web Push SHALL 兼容支持 Push API 的浏览器 |
| NFR-5.3 | 后端 SHALL 运行于 Python 3.10+（与现有基座一致） |
| NFR-5.4 | MySQL SHALL 使用 8.4 版本（与现有 compose.yml 一致），字符集 utf8mb4 |

### NFR-6 数据

| 编号 | 需求描述 |
|------|----------|
| NFR-6.1 | 数据库字符集 SHALL 为 utf8mb4，排序规则 utf8mb4_unicode_ci（支持 emoji 和中文） |
| NFR-6.2 | 引擎 SHALL 使用 InnoDB（支持事务和外键） |
| NFR-6.3 | 对话记录 tokens_used SHALL 被记录，用于 LLM 成本核算 |
| NFR-6.4 | Redis SHALL 作为 Agent 对话记忆基础设施，首版存储最近 N 轮上下文 |

---

### NFR-7 成本控制

| 编号 | 需求描述 |
|------|----------|
| NFR-7.1 | LLM Token 消耗 SHALL 被记录（conversations.tokens_used），支持按日/月统计费用 |
| NFR-7.2 | 系统 SHALL 支持配置 LLM 月度预算上限，接近上限时告警 |
| NFR-7.3 | iLink Bot 接入 SHALL 与核心业务解耦：即使 iLink 不可用，PWA + API 核心功能仍可正常使用 |


## 四、决策记录

以下问题已确认，作为开发约束：

| # | 问题 | 决策 |
|---|------|------|
| Q1 | 伴侣关联方式 | **邀请码**：A 生成邀请码，B 在 PWA 输入后绑定 |
| Q2 | iLink 接入状态 | **待调研**：先解耦开发，PWA + API 核心先跑通 |
| Q3 | LLM 预算 | **有预算控制**：记录 Token 消耗，支持月度上限告警 |
| Q4 | 用户注册方式 | **用户名+密码**：最简单流程，无邮箱验证 |
| Q5 | 数据共享粒度 | **全量共享**：建立伴侣关系后自动共享所有待办和日程 |
| Q6 | 推送渠道 | **Web Push 优先**：后续可扩展原生推送 |
| Q7 | 离线策略 | **完整离线**：离线创建+查看，联网自动同步 |
| Q8 | 部署环境 | **本地 Docker 先开发**：生产部署延后 |
